# 수입레이더 로드맵

롤모델: 포켓커스텀(pocket-custom.co.kr) — 관세사가 직접 만든 HS코드·통관 정보 플랫폼.
전체 코드를 그대로 가져올 수는 없으니(저작권), 기능 구성과 정보 구조만 참고하고
색·폰트·문구·UI 디테일·데이터 파이프라인은 전부 우리 것으로 새로 짠다.

## 목표 기능맵 (포켓커스텀 기준)

포켓커스텀이 제공하는 기능을 원본 그대로 옮기지 않고, 우리 데이터 파이프라인으로
현실적으로 만들 수 있는지 기준으로 재구성했다.

```
HS CODE
 ├─ 빠른 HS CODE 분류 (제품명 → HS코드 AI 추천)
 ├─ 분류표 보기 (HS코드 전체 탐색/검색)
 └─ 코드원 (배대지 API 연동)               ← B2B 전용, 후순위

도구
 ├─ 수입 관부가세 계산기
 └─ KC 인증대상 확인

식품검역
 ├─ 식품유형 확인
 ├─ 정밀검사비용 확인
 ├─ 영양성분 입력대상 확인
 └─ 영양성분 퍼센트 계산

수입식품 검역 의뢰
 └─ 가공식품 의뢰 (JNB관세사무소로 리드 전달)
```

## 단계별 로드맵

### Phase 1 — 데이터만 있으면 바로 되는 것 (다음 순서)

**1-1. HS코드 분류표 보기**
- 필요 데이터: 관세청 HS코드 마스터 목록(품목명·류/호 전체, 약 16,000여 개) — 새 data.go.kr API 연동 필요
- 파이프라인: 다른 3개 소스와 동일하게 GitHub Actions 주간 배치로 전체 수집 (이 API는 벌크 목록 조회가 가능할 것으로 예상 — 확인 필요)
- UI: 검색 가능한 표(현재 수입요건 표와 같은 스타일), 류(2자리)/호(4자리) 단위 트리 탐색 가능하면 더 좋음
- 애니메이션: 불필요 (데이터 탐색 화면은 정보 밀도가 우선)

**1-2. 관세율 조회 + 통관비용 계산기**
- 필요 데이터: 관세청 관세율표 API(실행세율·기본세율·FTA협정세율) — 새 데이터소스
- 파이프라인: 1-1과 함께 또는 별도 주간 배치
- UI: 계산기 폼(신고가격·운임·보험료 입력) → 결과 카드(관세+부가세+합계)
- 애니메이션: 결과 카드 등장 시 우리 시그니처인 "도장 찍힘" 모션 재사용 — 계산 완료를 도장처럼 확정짓는 느낌으로 일관성 유지

### Phase 2 — 중간 난이도

**2-1. AI 품목분류 (제품명 → HS코드 추천)**
- 이미 연동된 Claude API 활용 가능. 포켓커스텀처럼 2만 건 분류사례 학습 모델 수준은 아니지만, 관세율표 통칙+주규정을 프롬프트에 포함한 v1으로 시작 가능
- UI: 제품명·재질·용도 입력 → AI 후보 HS코드 2~3개 제시 (신뢰도 낮음을 명시)
- 애니메이션: 후보 도출 과정을 단계별로 보여주면 신뢰감 상승 (포켓커스텀도 "부/류 후보→호 후보→최종 호" 단계 연출을 씀)

**2-2. KC 인증대상 확인**
- 필요 데이터: 국가기술표준원 KC 인증대상 품목 API — 새 데이터소스
- UI: 현재 검색과 동일한 패턴 재사용 (HS코드 또는 품목명 입력 → 대상 여부 배지)

**2-3. 식품유형 확인**
- 필요 데이터: 식약처 식품공전 식품유형 기준 — 새 데이터소스
- UI: 라벨/성분표 텍스트 입력 → 해당 식품유형 후보 제시

### Phase 3 — 장기 과제

- **영양성분 입력대상/퍼센트 계산**: 규칙 기반 계산기, 데이터보다 로직이 복잡함
- **정밀검사비용 확인**: 식약처 고시 수수료 표 — 정적 참고표로 시작 가능
- **가공식품 검역 의뢰(리드 폼)**: Supabase에 문의 테이블 만들고 JNB관세사무소로 이메일 알림 연동. 실제 사업 액션이라 신중하게 설계 필요(개인정보 처리방침 등)
- **식품 라벨 사진 진단**: Claude Vision + 식품유형/원료 데이터 결합, 스코프가 가장 크다. Phase 1·2가 끝난 뒤 재검토

## 애니메이션 원칙 (일관성 유지용)

우리 시그니처는 통관 도장이다. 새 기능을 붙일 때마다 아무 모션이나 쓰지 말고:
- **결과가 확정되는 순간**(계산 완료, 조회 완료, 접수 완료)에만 도장 모션 사용
- **탐색/입력 중**에는 모션 최소화 (분류표 탐색, 폼 입력 등)
- `prefers-reduced-motion` 항상 존중 (이미 적용됨)
